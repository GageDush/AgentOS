## Problems to Fix

### Problem 1 — Context is siloed per AI tool
Each AI starts cold every session. You're maintaining `AGENTOS_MASTER_CODEX_PROMPT.md` and `CODEX_REPAIR_PROMPT.md` as manual workarounds. The wiki system exists specifically to solve this but the sync is manual and incomplete.

**Fix:** Set `AGENTOS_CURSOR_WIKI_SYNC=true` in `.env` — this alone makes Cursor sessions auto-feed the wiki on a timer. For ChatGPT exports: drop files in `.agentos/imports/chatgpt/` then run `pnpm wiki:sync-chatgpt`. For Claude Design: create `.agentos/imports/claude-design/` as a drop folder and add a wiki sync script for it.

---

### Problem 2 — Stack needs manual babysitting to start
The stack isn't reliably self-healing. `repair-cloudflare-tunnel.ps1` exists but isn't automated. The autostart script exists but may not be installed.

**Fix (3 steps):**
1. Run `pnpm install:autostart` once to register the stack with Windows Task Scheduler.
2. Add a scheduled task that runs `pnpm tunnel:repair` every 5 minutes (simple Task Scheduler entry).
3. Verify: `pnpm stack:status` should show all services green after reboot without manual intervention.

---

### Problem 3 — Duplicate / fragmented state stores
`.agentos/state/` has both `agentos-local.db` (SQLite) AND `agentos-local.json` (JSON mirror) plus a dozen other JSON files. Docker-compose has Postgres defined but not wired up. This creates "which is the truth?" confusion.

**Fix:**
- Delete `agentos-local.json` once confident SQLite is stable — it's a legacy mirror.
- Consolidate ad-hoc JSON files (`discord-registry.json`, `mission-route-cache.json`, etc.) into SQLite tables over time.
- Defer Postgres migration until you're ready to go multi-machine or cloud-hosted.

---

### Problem 4 — Feature flag sprawl
15+ `FEATURE_*` and `AGENTOS_*` flags in `.env.example`, many defaulting to `false`. Dead code paths are being maintained across every AI session, which causes AI tools to waste tokens reasoning about features that are off.

**Fix:** Do a quarterly flag audit. Any flag that's been `false` for 2+ months with no roadmap entry should be removed. Current candidates to prune: `FEATURE_CLOUD_PROVIDERS=false`, `FEATURE_TOOL_EXECUTION=false`, `FEATURE_LLM_TOOL_LOOP=false`, `FEATURE_TASK_SPAWN=false`.

---

### Problem 5 — Key files living outside the repo
`Desktop/system-routing-schematic.md` is a key architecture doc but isn't committed. The `cursor_chat_agentos_project_analysis_and_ove.json` export on the Desktop hasn't been wiki-synced. Landing page backups are loose HTMLs.

**Fix:**
- Move `Desktop/system-routing-schematic.md` → `docs/architecture/system-routing-schematic.md` and commit.
- Run `pnpm wiki:sync-chatgpt` after dropping the JSON export into `.agentos/imports/chatgpt/`.
- Landing page HTML backups → `docs/design/landing-pages/`.

---

### Problem 6 — Multi-AI handoff is implicit
When you switch from Cursor to Codex to Claude, each one needs to re-derive what the project is. The wiki helps but isn't the first thing any AI reads.

**Fix: Create a `CLAUDE.md` at the repo root** (and `CODEX.md` if Codex doesn't already read `AGENTS.md`) — each a 1-page "read this first" that points to wiki articles and lists the active sprint. This is the standardized version of your `AGENTOS_MASTER_CODEX_PROMPT.md`:

```
CLAUDE.md     → points to wiki + key docs, Claude-specific notes
CODEX.md      → same for Codex
CURSOR.md     → same for Cursor  
```

These are cheap to maintain and eliminate the cold-start problem.

---

### Problem 7 — No CI running on GitHub
`.gitlab-ci.yml` exists (GitLab conventions) but the repo is on GitHub (`github.com/GageDush/AgentOS`). No GitHub Actions workflow exists, so the release-manager agent has no CI signal to rely on.

**Fix:** Add `.github/workflows/ci.yml` that runs `pnpm typecheck && pnpm test` on push. The release-manager agent already uses `gh` CLI — CI green/red gives it real data.

---

### Problem 8 — Discord is a single point of failure for auth
If your Discord bot token expires or the bot goes down, nobody can log into the Command Center. Auth, approvals, and notifications all route through Discord.

**Fix (short-term):** Add a `DISCORD_BOT_TOKEN` health check to the `/health` endpoint and surface it in the Forge nav bar. Long-term: activate the "trusted device" local fallback login — the wireframe for this already exists in `docs/`.

## One-Line Summary

You have a sophisticated, well-architected local AI ops platform — the main drag is that each AI tool works in isolation and the glue (wiki sync, autostart, tunnel health) requires manual intervention. Fixing the auto-sync and cold-start problem will eliminate 80% of the babysitting without changing the architecture at all.
